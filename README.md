# API Организационной Структуры

Бэкенд-приложение на Go для управления древовидной структурой подразделений компании и её сотрудниками. Проект разработан с учетом требований чистоты кода, модульности и строгой изоляции бизнес-логики.

## 🛠 Технологический стек

* **Язык разработки:** Go 1.24 (использование стандартной библиотеки `net/http`)
* **База данных:** PostgreSQL 16
* **ORM:** GORM
* **Миграции:** Goose
* **Контейнеризация:** Docker / Docker Compose

---

## 🚀 Инструкция по запуску

Приложение полностью контейнеризировано. Для запуска проекта вам понадобится только установленный Docker Desktop.

1. **Клонируйте репозиторий и перейдите в папку проекта:**
```bash
git clone <ссылка_на_ваш_репозиторий>
cd org-structure-api

```


2. **Запустите сборку и старт контейнеров:**
```bash
docker compose up --build

```


**Что происходит при запуске:**
* Поднимается контейнер с базой данных PostgreSQL.
* Механизм `healthcheck` контролирует готовность СУБД к приему подключений.
* Утилита `goose` автоматически накатывает SQL-миграции из папки `/migrations` (создаются таблицы, индексы, уникальные ограничения и внешние ключи).
* Go-приложение собирается через оптимальный `Multi-stage build` (итоговый образ базируется на легковесном `alpine`) и запускает HTTP-сервер на порту `:8080`.


3. **Проверка работоспособности:**
Выполните тестовый запрос для проверки статуса сервера:
```bash
curl http://localhost:8080/health

```


Ожидаемый ответ: `{"status":"OK","message":"Server is running"}`

## 🗺 Реализованные API Эндпоинты

### 🏢 Подразделения (Departments)

* **Создать подразделение**
* `POST /departments/`
* Тело запроса (JSON): `{"name": "Backend", "parent_id": null}`
* *Логика:* Имя триммится от пробелов, проверяется уникальность имени на одном уровне вложенности.


* **Получить подразделение и его структуру**
* `GET /departments/{id}`
* Query-параметры: `depth` (глубина дерева от 1 до 5, по умолчанию 1), `include_employees` (включение списка сотрудников, по умолчанию true).
* *Логика:* Динамически строит и подгружает дерево до указанной глубины. Сотрудники сортируются по алфавиту.


* **Переместить / Переименовать подразделение**
* `PATCH /departments/{id}`
* Тело запроса (JSON): `{"name": "New Name", "parent_id": 3}` (все поля опциональны).
* *Логика:* Встроена рекурсивная проверка на циклы в дереве (CTE-запрос). Нельзя переместить департамент внутрь самого себя или своего же поддерева (возвращает `409 Conflict`).


* **Удалить подразделение**
* `DELETE /departments/{id}`
* Query-параметры: `mode` (`cascade` или `reassign`). Если выбран `reassign`, обязателен параметр `reassign_to_department_id`.
* *Логика:* В режиме `cascade` удаляется ветка дерева и все сотрудники в ней. В режиме `reassign` сотрудники атомарно переводятся в указанный департамент, дочерние структуры поднимаются на уровень выше, а целевой департамент удаляется. Все операции защищены БД-транзакциями.



### 🧑‍💻 Сотрудники (Employees)

* **Создать сотрудника в подразделении**
* `POST /departments/{id}/employees/`
* Тело запроса (JSON): `{"full_name": "Иван Иванов", "position": "Go Developer", "hired_at": "2026-05-24T00:00:00Z"}`
* *Логика:* Запрещено создание сотрудника в несуществующем подразделении (`404 Not Found`). Поля валидируются на длину (1..200 символов).

## 🏗 Архитектура проекта

Код строго разделен на слои в соответствии с принципами чистой/слоистой архитектуры (`Clean/Layered Architecture`), что исключает смешивание бизнес-логики с транспортным кодом или запросами к БД:

* `cmd/api/main.go` — Точка входа. Отвечает за чтение конфигурации, безопасное подключение к инфраструктуре, сборку зависимостей (Dependency Injection) и реализацию `Graceful Shutdown` (плавного завершения работы сервера).
* `config/` — Чтение настроек приложения из переменных окружения (`.env`), обеспечение принципа Fail-Fast.
* `internal/domain/` — Ядро системы. Содержит структуры бизнес-сущностей и интерфейсы-контракты репозиториев и юзкейсов. Не зависит от внешних библиотек и ORM.
* `internal/usecase/` — Слой бизнес-логики. Здесь инкапсулированы правила валидации, тримминг строк, логика проверки древовидных циклов и обработка стратегий удаления.
* `internal/repository/` — Инфраструктурный слой. Реализует интерфейсы домена с использованием GORM и PostgreSQL. Отвечает за рекурсивные SQL-запросы (CTE), транзакции и оптимизацию пула соединений.
* `internal/handler/` — Транспортный слой. Работает с протоколом HTTP, маршрутизацией (`net/http`), парсингом JSON, валидацией типов и маппингом внутренних ошибок в соответствующие HTTP-статусы.
* `migrations/` — SQL-файлы миграций для контроля схемы данных на уровне СУБД.
